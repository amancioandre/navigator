### Component Responsibilities

| Component | Responsibility | Implementation |
|-----------|----------------|----------------|
| **mkdocs.yml** | Central doc configuration: theme, nav, plugins, extensions | YAML file at project root |
| **docs/** | Markdown source files for all documentation pages | Directory of .md files organized by section |
| **mkdocs-typer2** | Auto-generates CLI reference from live Typer `app` object | MkDocs plugin; reads `navigator.cli:app` at build time |
| **mkdocstrings** | Renders API docs from Python docstrings (Pydantic models, config) | MkDocs plugin with `mkdocstrings-python` handler |
| **Material theme** | Responsive theme with search, dark mode, nav tabs, admonitions | `mkdocs-material` package |
| **site/** | Built static HTML output | Generated by `mkdocs build`, gitignored |
| **README.md** | Project entry point on GitHub, standalone from docs site | Handwritten, links to docs site |

### What Changes in Existing Files

| File | Change | Why |
|------|--------|-----|
| `pyproject.toml` | Add `[dependency-groups] docs = [...]` | Isolate doc dependencies from runtime and dev deps |
| `.gitignore` | Add `site/` | MkDocs build output should not be committed |
| No source code changes | -- | Documentation reads from existing code; no modifications needed |

### Structure Rationale

- **`docs/` at project root:** MkDocs convention. Keeps doc source separate from `src/`. The `mkdocs.yml` `docs_dir` defaults to `docs/`.
- **`mkdocs.yml` at project root:** Required location. MkDocs expects it adjacent to `docs/`.
- **`docs/guides/` mirrors features:** One guide per major capability matches what users search for. Each guide is self-contained.
- **`docs/reference/cli.md` uses mkdocs-typer2:** Single file with a directive that auto-generates the full CLI tree. No manual maintenance.
- **No `docs/api/` folder initially:** Navigator is a CLI tool, not a library. Users interact via commands, not Python imports. API docs for internal modules are unnecessary for v1.1.

## Integration Points with Existing Architecture

### pyproject.toml Changes

```toml
[dependency-groups]
dev = [
    "pytest>=9.0",
    "ruff>=0.15",
]
docs = [
    "mkdocs>=1.6",
    "mkdocs-material>=9.6",
    "mkdocs-typer2>=1.0",
    "mkdocstrings[python]>=0.29",
]
```

Use `dependency-groups` (not `optional-dependencies`) because doc tools are development-only and should never be installed by end users. `uv sync --group docs` installs them.

### CLI Reference Auto-Generation

The `docs/reference/cli.md` file contains a mkdocs-typer2 directive that reads the live Typer app:

```markdown
# CLI Reference

::: mkdocs-typer2
    :module: navigator.cli
    :name: navigator
```

At build time, mkdocs-typer2 imports `navigator.cli`, walks the Typer/Click command tree, and renders full documentation for every command, subcommand, argument, and option. This means:

1. The navigator package must be importable at build time (install with `uv sync` first)
2. All Typer `help` strings and docstrings become the CLI reference automatically
3. Adding new CLI commands automatically updates the reference on next build

## Data Flow

### Documentation Build Flow

```
Developer runs: uv run mkdocs build (or mkdocs serve)
    │
    ▼
MkDocs reads mkdocs.yml
    │
    ├──▶ Scans docs/ for .md files
    │
    ├──▶ mkdocs-typer2 plugin activates
    │    └──▶ Imports navigator.cli:app
    │         └──▶ Walks Click command tree
    │              └──▶ Generates CLI reference HTML
    │
    ├──▶ mkdocstrings plugin activates (if any ::: directives found)
    │    └──▶ Griffe parses src/navigator/ source files
    │         └──▶ Extracts docstrings + type annotations
    │              └──▶ Renders API documentation
    │
    ├──▶ Material theme renders all pages
    │    └──▶ Adds search index, navigation, styling
    │
    └──▶ Outputs to site/ directory (static HTML + JS + CSS)
```

### Local Development Flow

```
uv sync --group docs          # Install doc dependencies
    │
    ▼
uv run mkdocs serve           # Start dev server on localhost:8000
    │                          # Auto-reloads on docs/ changes
    ▼
Edit docs/*.md files           # Saved changes appear instantly
    │
    ▼
uv run mkdocs build --strict  # Final validation before commit
                               # --strict fails on warnings (broken links, etc.)
```

### Key Data Flows

1. **Code to CLI Reference:** Typer app help strings and docstrings flow through mkdocs-typer2 at build time into rendered HTML. No intermediate files.
2. **Markdown to Site:** Hand-written guide .md files are rendered through Material theme with syntax highlighting, admonitions, and navigation.
3. **README.md is independent:** README.md is not part of the MkDocs site. It lives at project root for GitHub/PyPI display and links to the docs site.

## Architectural Patterns

### Pattern 1: Docs as Code

**What:** Documentation lives in the same repository as source code, versioned together, reviewed together.
**When to use:** Always for developer tools and CLI applications.
**Trade-offs:** Docs can drift from code if not auto-generated. Mitigated here by mkdocs-typer2 pulling CLI docs from the live app.

### Pattern 2: Auto-Generated Reference, Hand-Written Guides

**What:** CLI reference is auto-generated from Typer decorators and help strings. Feature guides are hand-written narrative documentation explaining concepts and workflows.
**When to use:** When you have both a mechanical API surface (commands, options) and conceptual knowledge (how features work together, best practices).
**Trade-offs:** Requires maintaining guide content manually. But guides provide the "why" and "how" that auto-generated docs cannot.

**Boundary rule:** If it is a fact about a command's arguments, it belongs in help strings (auto-generated). If it explains a workflow or concept, it belongs in a guide (hand-written).

### Pattern 3: Dependency Group Isolation

**What:** Doc dependencies are in their own dependency group, separate from runtime and dev.
**When to use:** When doc tools are heavy (MkDocs + Material + plugins pull in many transitive deps) and should not pollute the dev environment unless explicitly requested.
**Trade-offs:** Developers must run `uv sync --group docs` explicitly. This is intentional -- most development does not require doc tools.

```toml
# pyproject.toml
[dependency-groups]
dev = ["pytest>=9.0", "ruff>=0.15"]
docs = ["mkdocs>=1.6", "mkdocs-material>=9.6", "mkdocs-typer2>=1.0", "mkdocstrings[python]>=0.29"]
```

## Deployment Considerations

### Local Only (Recommended for v1.1)

Navigator is a private tool. Documentation serves the developer (you) and Claude Code agents.

```bash
uv run mkdocs serve              # Live preview at localhost:8000
uv run mkdocs build              # Build to site/ for offline browsing
```

No deployment infrastructure needed. `mkdocs serve` is sufficient for a single-user private project.

### GitHub Pages (Optional, if project goes public later)

If Navigator is ever published:

```bash
uv run mkdocs gh-deploy          # Builds and pushes to gh-pages branch
```

Or via GitHub Actions:

```yaml
# .github/workflows/docs.yml
name: Deploy docs
on:
  push:
    branches: [main]
    paths: ['docs/**', 'mkdocs.yml', 'src/navigator/**']
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: astral-sh/setup-uv@v5
      - run: uv sync --group docs
      - run: uv run mkdocs gh-deploy --force
```

**Recommendation:** Do not set up deployment for v1.1. Build locally. Add GitHub Pages deployment only if/when the project becomes public.

## Anti-Patterns

### Anti-Pattern 1: Duplicating CLI Help in Markdown

**What people do:** Manually write CLI reference docs that mirror `--help` output.
**Why it's wrong:** Drifts instantly when commands change. Double maintenance burden.
**Do this instead:** Use mkdocs-typer2 to auto-generate CLI reference from the live Typer app. Improve docs by improving help strings in the source code.

### Anti-Pattern 2: Using mkdocstrings for CLI Reference

**What people do:** Point mkdocstrings at CLI module functions to document the CLI.
**Why it's wrong:** mkdocstrings documents Python APIs (classes, functions, parameters). CLI users interact with commands and options, not Python functions. The output format is wrong for the audience.
**Do this instead:** mkdocs-typer2 for CLI reference (renders commands/options). mkdocstrings only if you need to document Python APIs (e.g., Pydantic model schemas for advanced users).

### Anti-Pattern 3: Giant Single-Page Documentation

**What people do:** Put everything in README.md or a single docs page.
**Why it's wrong:** Unusable for reference. Users cannot find specific information. Search does not help when everything is one page.
**Do this instead:** Separate pages per topic. MkDocs search handles discovery. README.md is an entry point with links, not a comprehensive manual.

### Anti-Pattern 4: Docs Dependency in Main Dependencies

**What people do:** Add mkdocs to `[project.dependencies]` or the `dev` dependency group.
**Why it's wrong:** MkDocs + Material + plugins add 30+ transitive dependencies. Users installing navigator should not download documentation tooling. Even dev installs do not need it unless actively writing docs.
**Do this instead:** Separate `docs` dependency group. Install only when needed: `uv sync --group docs`.

## Build Order for Documentation Phases

Dependencies between documentation components dictate implementation sequence:

```
Phase 1: Foundation
    mkdocs.yml (configuration)
    └─▶ docs/index.md (homepage)
    └─▶ pyproject.toml docs dependency group
    └─▶ .gitignore update (site/)
    └─▶ Verify: uv sync --group docs && uv run mkdocs serve works

Phase 2: CLI Reference (auto-generated)
    docs/reference/cli.md with mkdocs-typer2 directive
    └─▶ Depends on: Phase 1 (mkdocs.yml with plugin configured)
    └─▶ Depends on: navigator package importable (uv sync)
    └─▶ May require improving help strings in src/navigator/cli.py

Phase 3: Getting Started
    docs/getting-started/installation.md
    docs/getting-started/quickstart.md
    └─▶ No code dependencies, purely hand-written
    └─▶ References CLI commands (benefits from Phase 2 being done for cross-checking)

Phase 4: Feature Guides
    docs/guides/scheduling.md
    docs/guides/watching.md
    docs/guides/chaining.md
    docs/guides/secrets.md
    docs/guides/namespaces.md
    docs/guides/systemd.md
    └─▶ Each guide is independent; can be written in any order
    └─▶ References CLI commands documented in Phase 2

Phase 5: README.md
    README.md at project root
    └─▶ Depends on: installation docs (Phase 3) for consistent instructions
    └─▶ Links to docs site for details
    └─▶ Standalone: does not use MkDocs, rendered by GitHub/PyPI directly

Phase 6: Polish
    Review navigation structure
    Run mkdocs build --strict to catch broken links
    Add any missing cross-references between guides
```

Each phase produces a usable increment. After Phase 1, you have a working doc site skeleton. After Phase 2, the most valuable page (CLI reference) is live and auto-maintained. Phases 3-4 add narrative content. Phase 5 handles the GitHub-facing README.
