# Phase 9: Daemon and systemd - Context

**Gathered:** 2026-03-24
**Status:** Ready for planning

<domain>
## Phase Boundary

Deliver persistent daemon support — the watcher daemon runs as a systemd user service that survives reboots. `navigator install-service` generates and installs unit files. `navigator service` wraps systemctl for easy management. After this phase, watchers automatically restart on failure and resume on boot.

</domain>

<decisions>
## Implementation Decisions

### Service Configuration
- **D-01:** systemd user service at `~/.config/systemd/user/navigator.service` — no root required.
- **D-02:** Unit file generated by `navigator install-service` — dynamically resolves navigator binary path and config.
- **D-03:** Restart policy: `Restart=on-failure`, `RestartSec=5`.
- **D-04:** `WantedBy=default.target` — starts on user login.

### Daemon Behavior
- **D-05:** Daemon entry point: `navigator daemon` subcommand — runs watcher daemon in foreground (systemd manages background).
- **D-06:** `navigator install-service` enables `loginctl enable-linger` for the user — service runs even when not logged in.
- **D-07:** `navigator service status/start/stop/restart` — wraps `systemctl --user` commands for convenience.

### Claude's Discretion
- Module organization (single `service.py` or daemon logic in watcher.py)
- Unit file template format (Python string template vs Jinja2 vs f-string)
- Whether to add `navigator uninstall-service` for cleanup
- Test approach for systemd operations (mock subprocess for systemctl calls)
- Whether `navigator daemon` is the existing `watch --start` or a new subcommand

</decisions>
